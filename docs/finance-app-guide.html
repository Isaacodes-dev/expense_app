1. Project Overview

The Finance App is a personal finance management tool that helps users track income, expenses, budgets, and savings goals. It is built as a single-page application (SPA) with a decoupled backend API.

2. Backend Improvements

2.1 BelongsToUser Global Scope

File: app/Models/Concerns/BelongsToUser.php

Problem: Every controller query manually filtered with ->where('user_id', $request->user()->id). This is repetitive and, more critically, if a developer forgets the filter in a new query, it creates a data leak where users can see each other's records.

Solution: A reusable trait that adds a global scope to automatically filter all queries by Auth::id(). It also auto-sets user_id on newly created models.

trait BelongsToUser
{
    protected static function bootBelongsToUser(): void
    {
        static::addGlobalScope('user', function (Builder $query) {
            if (Auth::check()) {
                $query->where($query->getModel()->getTable() . '.user_id', Auth::id());
            }
        });

        static::creating(function (Model $model) {
            if (Auth::check() && ! $model->user_id) {
                $model->user_id = Auth::id();
            }
        });
    }
}

Applied to: Category, Income, Expense, Budget, SavingsGoal

Impact: Controllers no longer need to manually pass user_id during creation or filter by it during queries. For example, creating an expense simplifies from:

// Before
$expense = Expense::create([
    'user_id' => $request->user()->id,
    ...$request->validated(),
]);

// After
$expense = Expense::create($request->validated());

2.2 SavingsGoalStatus Enum

File: app/Enums/SavingsGoalStatus.php

Problem: The status field on SavingsGoal was a bare string with no model-level type safety. Invalid values like "pending" could be written without error.

Solution: A PHP 8.3 backed enum, cast on the model:

enum SavingsGoalStatus: string
{
    case Active = 'active';
    case Completed = 'completed';
    case Cancelled = 'cancelled';
}

// In SavingsGoal model:
protected $casts = [
    'status' => SavingsGoalStatus::class,
];

Now assigning an invalid status throws a ValueError instead of silently corrupting data.

2.3 Controller Cleanup

Removed unnecessary fresh() calls

Affected: BudgetController, ExpenseController, CategoryController, SavingsGoalController, IncomeController, SavingsContributionController

After calling $model->update(), the model instance is already synced. Calling $model->fresh() fires an extra SELECT query for no benefit.

// Before (extra query)
return new BudgetResource($budget->fresh()->load('category:id,name'));

// After (no extra query)
$budget->load('category:id,name');
return new BudgetResource($budget);

Extracted duplicate expense aggregation

Both BudgetController and DashboardService had identical queries to sum expenses by category. Extracted to a shared static method:

// BudgetController
public static function getActualByCategory(int $month, int $year): Collection
{
    return Expense::whereMonth('date', $month)
        ->whereYear('date', $year)
        ->selectRaw('category_id, SUM(amount) as total')
        ->groupBy('category_id')
        ->pluck('total', 'category_id');
}

// DashboardService now calls:
$actualByCategory = BudgetController::getActualByCategory($month, $year);

2.4 Form Request Authorization

Problem: All Form Requests returned authorize() → true while controllers separately called $this->authorize(). This split responsibility and meant validation ran before authorization—wasting resources on invalid requests.

Solution: Update requests now handle authorization via policies directly:

// UpdateExpenseRequest
public function authorize(): bool
{
    return $this->user()->can('update', $this->route('expense'));
}

// Controller no longer needs:
// $this->authorize('update', $expense);  ← removed

Applied to: UpdateCategoryRequest, UpdateExpenseRequest, UpdateIncomeRequest, UpdateBudgetRequest, UpdateSavingsGoalRequest

2.6 Rate Limiting & Route Security

File: routes/api.php

  • Auth endpoints (login, register): throttle:6,1 — 6 requests per minute. Prevents brute-force attacks.
  • Authenticated routes: throttle:60,1 — 60 requests per minute per user.
  • Scoped nested resources: savings-goals.contributions now uses ->scoped() which ensures a contribution's parent savings goal matches the route parameter, preventing access to contributions via a different goal's URL.

2.7 Exception Handling

File: bootstrap/app.php

Added a custom renderer for NotFoundHttpException on API routes. Previously, a 404 on /api/expenses/999 would return an HTML error page. Now it returns proper JSON:

{ "message": "Resource not found." }

3. Frontend Improvements

3.1 HTTP Error Interceptor

File: resources/js/services/http.ts

Problem: The Axios instance had no error handling. A 401 response (expired session) would silently fail, leaving the user on a broken page with no feedback.

Solution: Added a response interceptor and a 30-second request timeout:

StatusActionReason
401 UnauthorizedRedirect to login pageSession expired or user logged out elsewhere
419 CSRF MismatchReload the pageCSRF token expired; reload fetches a fresh token
Timeout (30s)Reject promisePrevents indefinite loading states

3.2 Router Guard Race Condition

File: resources/js/router/guards.ts

Problem: If the user navigated rapidly (e.g., clicked a link during initial page load), multiple concurrent fetchUser() calls could fire. This wasted requests and could cause the auth check to evaluate before the first fetch completed.

Solution: Deduplicates the fetch with a module-level promise reference:

let fetchingUser: Promise<void> | null = null;

router.beforeEach(async (to) => {
    if (!authStore.checked) {
        if (!fetchingUser) {
            fetchingUser = authStore.fetchUser();
        }
        await fetchingUser;  // All concurrent navigations wait on the same promise
    }
    // ... auth checks
});

3.3 Route Meta Types

File: resources/js/types/router.d.ts

Augments Vue Router's RouteMeta interface so that meta.requiresAuth, meta.guest, and meta.layout are type-checked. A typo like meta.requireAuth is now caught at compile time.

declare module 'vue-router' {
    interface RouteMeta {
        layout?: 'auth' | 'app';
        requiresAuth?: boolean;
        guest?: boolean;
    }
}

3.4 Error Boundary

File: resources/js/App.vue

Problem: If a lazy-loaded page component threw an error during render, the entire app would crash with a white screen and no recovery path.

Solution: Added onErrorCaptured at the root component. When an unhandled error occurs, a fallback UI is shown with a "Reload page" button instead of a blank screen.

3.5 Typed Error Handling

File: resources/js/utils/api-error.ts

Problem: Catch blocks used err: any, defeating TypeScript. ESLint had no-explicit-any disabled globally.

Solution: Created typed error utilities and changed the ESLint rule to warn:

// Type-safe error handling
import { getErrorMessage } from '@/utils/api-error';

try {
    await store.remove(id);
} catch (err: unknown) {
    notify.error('Error', getErrorMessage(err, 'Failed to delete.'));
}

Also removed the [key: string]: unknown index signature from the User type, which previously allowed any arbitrary property without a type error.

4.2 Dockerfile (Multi-Stage)

The Dockerfile uses two stages to produce a lean production image:

Stage 1: node-builder

  • Base: node:22-alpine
  • Copies package.json and package-lock.json first for layer caching—npm only re-installs when dependencies change.
  • Accepts VITE_APP_NAME, VITE_LOCALE, VITE_CURRENCY as build args (Vite inlines these at compile time).
  • Runs npm run build, outputting hashed assets to public/build/.

Stage 2: php

  • Base: php:8.3-fpm-alpine
  • Installs PHP extensions: pdo_pgsql, pgsql, bcmath, mbstring, xml, zip, curl, opcache.
  • OPcache tuned for production (validate_timestamps=0).
  • Composer installs with --no-dev --optimize-autoloader.
  • Copies built frontend assets from stage 1.
  • Optional Xdebug via --build-arg INSTALL_XDEBUG=true (excluded by default).
  • Runs as www-data (non-root) for security.
  • Entrypoint caches Laravel config/routes/views and runs migrations.
Why not generate APP_KEY in the Dockerfile? The APP_KEY encrypts sessions, cookies, and other sensitive data. Generating it on every container start would invalidate all existing sessions. It must be set once via .env and reused across deployments.

4.3 Nginx Configuration

File: docker/nginx.conf

Key routing rules:

PathHandlerCache
/build/*Static files (Vite)Cache-Control: public, immutable (content-hashed filenames)
/api/*FastCGI → PHP-FPMNone
/sanctum/csrf-cookieFastCGI → PHP-FPMNone (sets XSRF-TOKEN cookie)
/upFastCGI → PHP-FPMNone (health check)
/*try_files → /index.phpNone (SPA fallback for Vue Router)

Security headers applied globally: X-Frame-Options, X-Content-Type-Options, Referrer-Policy. Dotfiles are denied. Gzip is enabled for text-based assets.

No CORS headers needed. Since the Vue SPA and the API are served from the same Nginx on the same domain, all requests are same-origin. Sanctum uses first-party session cookies; no Access-Control-* headers are required.

4.4 Docker Compose (Production)

File: docker-compose.yml

Key design decisions:

  • Shared app-public volume — Nginx reads static files (public/build/, favicon, etc.) directly from this volume instead of proxying them through PHP-FPM. The :ro flag ensures Nginx cannot modify files.
  • Database healthcheck — The app service uses depends_on: db: condition: service_healthy. This ensures migrations in the entrypoint don't fail because PostgreSQL isn't ready yet.
  • Fail-fast on missing password — ${DB_PASSWORD:?DB_PASSWORD must be set} makes Docker Compose refuse to start if the password is not configured, preventing accidental deployment with an empty database password.
  • Queue worker — Uses the same Docker image but overrides CMD. The entrypoint detects that the command is not php-fpm and skips caching/migrations.

4.5 Docker Compose (Development)

File: docker-compose.override.yml

Automatically merged when running docker compose up (without -f).

  • Bind mount — .:/var/www/html maps host code into the container for live editing. An anonymous volume (/var/www/html/vendor) preserves the container's Linux-native vendor directory.
  • Entrypoint override — Skips config/route caching so .env changes take effect immediately.
  • Vite HMR — A node-dev service runs the Vite dev server on port 5173. The @vite Blade directive auto-detects this via the public/hot file.
  • Exposed DB port — PostgreSQL is accessible at localhost:5432 for GUI tools.

4.6 Deployment Guide

First-Time Setup

# 1. Copy and configure environment
cp .env.example .env
# Edit .env: set DB_HOST=db, DB_PASSWORD=<strong-password>

# 2. Build and start (production)
docker compose -f docker-compose.yml build
docker compose -f docker-compose.yml up -d

# 3. Generate application key
docker compose exec app php artisan key:generate --show
# Add output to .env as APP_KEY=base64:...
docker compose restart app

Development

# Start all services (auto-merges override)
docker compose up -d

# App:      http://localhost:8080
# Vite HMR: http://localhost:5173
# Database: localhost:5432

Production Deployment

# Build with custom env vars
docker compose -f docker-compose.yml build \
  --build-arg VITE_APP_NAME="My Finance App"

# Start (skips dev override)
docker compose -f docker-compose.yml up -d

# View logs
docker compose logs -f app
docker compose logs -f queue-worker

Common Operations

# Run migrations manually
docker compose exec app php artisan migrate

# Clear all caches
docker compose exec app php artisan optimize:clear

# Access PostgreSQL shell
docker compose exec db psql -U postgres -d finance_app

# Restart queue worker (after code changes)
docker compose restart queue-worker
